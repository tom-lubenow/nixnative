# Scanner Requirements

Requirements for the dependency scanner subsystem.

## Correctness

### Dependency File Parsing
- Parser must handle Make-format `.d` files correctly: escaped spaces (`\ `), escaped colons, escaped backslashes, multi-line continuations (`\\\n`)
- Parser must not silently discard dependencies on parse errors; surface failures explicitly
- Self-references (source file in its own dependency list) should be filtered out

### Failure Handling
- Compilation failures during scanning must be reported with file context, not silently ignored
- Missing `.d` files after scan indicate compilation failure and must not produce empty dependency sets

### Override Conflicts
- Header/source override collisions (same relative path from multiple tools) must warn or error, not silently resolve

### Path Normalization
- All relative paths must be consistently normalized (e.g., `./foo` vs `foo`) across scanner, manifest, and compilation
- Path handling must be unified to prevent mismatches between scanner-detected and compile-time paths

## Performance

### Parallelism
- File scanning should support parallel execution; sequential per-file compilation is a bottleneck for large projects

### Source Tree Handling
- Scanner sandbox should minimize copied files; copying entire source tree is expensive for large projects
- Consider copying only scanned files plus declared include directories

### IFD Considerations
- Import From Derivation blocks Nix evaluation; strict mode (pre-computed manifests) remains the CI-friendly path
- Scanner invocations should be cacheable and deterministic

## Security

### Flag Expansion
- All compiler flags derived from user input (defines, cxxFlags, includeDirs) must be properly escaped or quoted in shell context
- Define values must not allow shell metacharacter injection

### Path Validation
- Relative paths from tool overrides must be validated to prevent directory traversal (`../`)

## Flexibility

### Language Support
- Scanner interface should accommodate C, C++, and potentially other languages requiring dependency detection
- Per-file language detection or explicit language specification may be needed (`.c` vs `.cc`)

### Per-File Configuration
- Consider whether per-file flag overrides are needed (different includes/defines for specific files)

### Exclusion Patterns
- Consider support for excluding files/patterns from scanning (test files, third-party code)

## Schema & Compatibility

### Manifest Schema
- Schema versioning (`schema = N`) must have migration/compatibility strategy for future changes
- Schema should remain stable; breaking changes require major version bump

### Alternative Scan Methods
- Clang's `-MJ` produces JSON directly but is clang-specific; cross-compiler consistency (GCC, other compilers) must be weighed against correctness benefits
- Any compiler-specific features must have fallback paths or be gated by capability detection

## Maintainability

### Code Organization
- Include flag generation logic should be centralized; duplication across scanner/compile/utils risks divergence

### Error Context
- Scanner errors should identify: which file failed, what operation was attempted, actionable remediation

### Tool Integration
- Tool plugin ordering affects scanning (generated headers must exist before scanning files that include them); this dependency should be explicit or automatically resolved

## Edge Cases

The following must have defined behavior:
- Circular header includes
- Symlinked headers and source files
- Unicode/special characters in file paths
- Very long include paths (command line limits)
- Files with identical basenames in different directories
- Headers generated by tools not yet executed
